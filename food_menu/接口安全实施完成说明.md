## 🔐 1. 接口限流

### 功能说明

使用Guava RateLimiter实现基于令牌桶算法的接口限流，支持：
- 按IP限流
- 按用户限流
- 全局限流

### 技术实现

**新增文件：**
- `RateLimiter.java` - 限流注解
- `RateLimiterAspect.java` - 限流切面

**依赖：**
```xml
<dependency>
    <groupId>com.google.guava</groupId>
    <artifactId>guava</artifactId>
    <version>32.1.3-jre</version>
</dependency>
```

### 使用示例

```java
@RateLimiter(
    qps = 5,                    // 每秒最多5个请求
    timeout = 500,              // 等待令牌超时时间500ms
    message = "请求过于频繁",    // 限流提示信息
    limitType = LimitType.IP    // 按IP限流
)
@PostMapping("/login")
public Result<String> login(@RequestBody LoginDto loginDto) {
    // 业务逻辑
}
```

### 已添加限流的接口

| 接口 | QPS | 限流类型 | 说明 |
|------|-----|---------|------|
| `/user/sendcode` | 1 | IP | 验证码发送（防刷） |
| `/user/login` | 5 | IP | 登录（防暴力破解） |
| `/wx/user/sendcode` | 1 | IP | 小程序验证码（防刷） |
| `/wx/user/login` | 5 | IP | 小程序登录（防暴力破解） |
| `/wx/user/register` | 2 | IP | 注册（防批量注册） |
| `/order/submit` | 3 | USER | 订单提交（防刷单） |
| `/dish/comments` | 5 | USER | 评论发表（防刷评论） |

### 限流策略说明

**1. 验证码接口（QPS=1）**
- 最严格的限流
- 防止短信轰炸
- 按IP限流，一秒只能请求一次

**2. 登录接口（QPS=5）**
- 防止暴力破解
- 按IP限流，一秒最多5次尝试
- 配合账号锁定策略更佳

**3. 订单提交（QPS=3）**
- 防止恶意刷单
- 按用户限流，每个用户一秒最多3单
- 防止误操作

**4. 评论接口（QPS=5）**
- 防止刷评论
- 按用户限流
- 保护内容质量

## 🛡️ 2. 防重复提交

### 功能说明

使用Caffeine本地缓存实现防重复提交机制，在指定时间间隔内拒绝相同的请求。

### 技术实现

**新增文件：**
- `PreventDuplicateSubmit.java` - 防重复提交注解
- `PreventDuplicateSubmitAspect.java` - 防重复提交切面

**依赖：**
```xml
<dependency>
    <groupId>com.github.ben-manes.caffeine</groupId>
    <artifactId>caffeine</artifactId>
</dependency>
```

### 使用示例

```java
@PreventDuplicateSubmit(
    interval = 3000,              // 3秒内不能重复提交
    message = "订单已提交",        // 重复提交提示
    byUser = true                 // 基于用户维度
)
@PostMapping("/order/submit")
public Result<Long> submit(@RequestBody OrdersDto ordersDto) {
    // 业务逻辑
}
```

### 已添加防重复提交的接口

| 接口 | 间隔时间 | 维度 | 说明 |
|------|---------|------|------|
| `/order/submit` | 3秒 | 用户 | 防止重复下单 |
| `/wx/user/register` | 2秒 | IP | 防止重复注册 |
| `/dish/comments` | 2秒 | 用户 | 防止重复评论 |

### 工作原理

1. **首次请求：**
   - 生成唯一key（基于用户ID或IP + 方法名）
   - 记录请求时间到缓存
   - 执行业务逻辑

2. **重复请求：**
   - 检查缓存中的上次请求时间
   - 如果在间隔时间内，拒绝请求
   - 返回友好提示信息

3. **业务失败：**
   - 如果业务执行失败，移除缓存
   - 允许用户重试

## 🔒 3. SQL注入防护

### 检查结果

✅ **项目无SQL注入风险**

详细报告请查看：`SQL注入防护报告.md`

### 防护措施

1. **参数化查询**
   - 所有SQL使用 `#{}` 而非 `${}`
   - MyBatis预编译SQL

2. **MyBatis-Plus ORM**
   - 使用MyBatis-Plus的CRUD方法
   - 自动生成安全SQL

3. **输入验证**
   - Controller层参数校验
   - DTO数据传输对象

### 代码规范

```java
// ✅ 正确：参数化查询
@Select("SELECT * FROM user WHERE id = #{id}")
User getUserById(@Param("id") Long id);

// ❌ 错误：字符串拼接
@Select("SELECT * FROM user WHERE id = ${id}")
User getUserById(@Param("id") Long id);
```

## 🎯 配置说明

### 限流配置调优

根据实际业务情况调整QPS：

```java
// 高并发场景
@RateLimiter(qps = 100)  // 每秒100个请求

// 普通场景
@RateLimiter(qps = 10)   // 每秒10个请求

// 敏感操作
@RateLimiter(qps = 1)    // 每秒1个请求
```

### 防重复提交配置

根据业务特点调整间隔：

```java
// 快速操作（如评论）
@PreventDuplicateSubmit(interval = 2000)  // 2秒

// 重要操作（如订单）
@PreventDuplicateSubmit(interval = 5000)  // 5秒

// 超敏感操作（如支付）
@PreventDuplicateSubmit(interval = 10000) // 10秒
```

## ⚠️ 注意事项

### 1. 单机限流

当前实现是**单机限流**，如果部署多个实例：
- 限流效果会按实例倍增
- 例如：QPS=10 的接口，2个实例总QPS为20

**解决方案（可选）：**
- 使用Redis实现分布式限流
- 使用网关层限流（Nginx/Kong）

### 2. 缓存清理

防重复提交使用本地缓存：
- 重启应用会清空缓存
- 不影响业务逻辑

### 3. 异常处理

限流和防重复提交都会抛出RuntimeException：

```java
try {
    // 调用接口
} catch (RuntimeException e) {
    if (e.getMessage().contains("频繁")) {
        // 限流异常处理
    } else if (e.getMessage().contains("重复")) {
        // 防重复提交异常处理
    }
}
```

## 🔧 故障排查

### 问题1：限流不生效

**检查：**
1. 确认接口有 `@RateLimiter` 注解
2. 确认 `RateLimiterAspect` 被Spring扫描到
3. 检查日志是否有"创建限流器"信息

### 问题2：防重复提交误判

**可能原因：**
- 间隔时间设置过长
- 用户ID获取失败（匿名用户）

**解决：**
- 调整 `interval` 参数
- 改用 `byUser = false`（按IP）

### 问题3：性能问题

**检查：**
- Caffeine缓存大小是否合理
- 限流器数量是否过多（建议< 1000）

**优化：**
- 调整缓存大小
- 使用更粗粒度的限流（如全局限流）
