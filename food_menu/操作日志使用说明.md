# 操作日志使用说明

## 概述

本项目已集成完整的操作日志和审计日志功能，能够自动记录关键业务操作和管理员操作，方便系统审计和问题追溯。

## 功能特性

- ✅ 自动记录操作日志（基于AOP切面）
- ✅ 支持多种操作类型（新增、修改、删除、查询、登录等）
- ✅ 记录操作人信息（ID、姓名、类型）
- ✅ 记录请求参数和响应结果
- ✅ 敏感信息自动脱敏
- ✅ 记录IP地址和用户代理
- ✅ 记录执行时长
- ✅ 区分操作成功和失败
- ✅ 异步保存日志，不影响业务性能
- ✅ 提供日志查询接口

## 使用方法

### 1. 在Controller方法上添加@OperationLog注解

```java
@RestController
@RequestMapping("/order")
public class OrdersController {
    
    /**
     * 提交订单
     */
    @OperationLog(
        operationType = OperationLog.OperationType.INSERT,
        operationModule = "订单",
        operationDesc = "提交订单",
        recordParams = true,     // 记录请求参数
        recordResult = true      // 记录响应结果
    )
    @PostMapping("/submit")
    public Result<Long> submit(@RequestBody OrdersDto ordersDto) {
        // 业务代码
        ordersService.submit(ordersDto);
        return Result.success(ordersDto.getId());
    }
    
    /**
     * 更新订单状态
     */
    @OperationLog(
        operationType = OperationLog.OperationType.UPDATE,
        operationModule = "订单",
        operationDesc = "更新订单状态",
        recordParams = true
    )
    @PutMapping("/status")
    public Result<String> updateStatus(@RequestParam Long id, @RequestParam Integer status) {
        ordersService.updateStatus(id, status);
        return Result.success("更新成功");
    }
}
```

### 2. 操作类型说明

| 操作类型 | 说明 | 使用场景 |
|---------|------|---------|
| INSERT | 新增操作 | 创建订单、添加菜品、注册用户等 |
| UPDATE | 修改操作 | 更新信息、修改状态等 |
| DELETE | 删除操作 | 删除数据 |
| QUERY | 查询操作 | 敏感数据查询、导出等 |
| LOGIN | 登录操作 | 用户登录 |
| LOGOUT | 登出操作 | 用户登出 |
| EXPORT | 导出操作 | 数据导出 |
| IMPORT | 导入操作 | 数据导入 |
| AUDIT | 审核操作 | 审核订单、审核评论等 |
| GRANT | 授权操作 | 分配权限、角色等 |
| OTHER | 其他操作 | 其他业务操作 |

### 3. 注解参数说明

```java
@OperationLog(
    operationType = OperationType.INSERT,  // 操作类型（必填）
    operationModule = "订单",              // 操作模块（必填）
    operationDesc = "提交订单",            // 操作描述（必填）
    recordParams = true,                   // 是否记录请求参数，默认true
    recordResult = false                   // 是否记录响应结果，默认false（避免数据过大）
)
```

## 权限控制

### 角色权限说明

| 角色 | role值 | 权限说明 |
|------|--------|---------|
| 普通管理员 | 0 | 只能查看和操作自己家庭的操作日志 |
| 家庭管理员 | 1 | 只能查看和操作自己家庭的操作日志 |
| 超级管理员 | 2 | 可以查看所有操作日志，可按家庭ID筛选 |

### 数据隔离规则

1. **超级管理员 (role=2)**
   - ✅ 可以查看所有家庭的操作日志
   - ✅ 可以通过 `familyId` 参数筛选特定家庭的日志
   - ✅ 可以查看、删除所有日志

2. **普通管理员/家庭管理员 (role=0/1)**
   - ✅ 只能查看自己家庭的操作日志
   - ❌ 无法查看其他家庭的日志
   - ✅ 只能删除自己家庭的日志
   - ❌ 无法通过 `familyId` 参数查看其他家庭的日志（会被自动过滤）

### 安全说明

- 所有接口都需要在请求头中携带 `Authorization: Bearer {token}`
- 系统会自动从Token中解析用户角色和家庭ID
- 如果Token无效或无法解析，将返回空结果或权限错误
- 权限检查在服务端进行，前端无法绕过

## 查询操作日志

### 1. 分页查询操作日志

**接口地址**：`GET /admin/operationLog/page`

**请求头**：
```
Authorization: Bearer {token}
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | int | 是 | 页码（默认1） |
| pageSize | int | 是 | 每页大小（默认10） |
| operationType | string | 否 | 操作类型 |
| operationModule | string | 否 | 操作模块 |
| operatorId | long | 否 | 操作人ID |
| operatorName | string | 否 | 操作人姓名（模糊查询） |
| operatorType | string | 否 | 操作人类型 |
| status | int | 否 | 状态（0失败 1成功） |
| familyId | long | 否 | 家庭ID（仅超级管理员可用，用于筛选特定家庭） |
| startTime | string | 否 | 开始时间（格式：yyyy-MM-dd HH:mm:ss） |
| endTime | string | 否 | 结束时间（格式：yyyy-MM-dd HH:mm:ss） |
| ipAddress | string | 否 | IP地址 |

**权限说明**：
- 超级管理员：可以查看所有日志，可通过 `familyId` 筛选
- 普通管理员/家庭管理员：只能查看自己家庭的日志，`familyId` 参数会被忽略

**请求示例**：

```bash
# 查询所有操作日志（超级管理员查看所有，普通管理员查看自己家庭）
GET /admin/operationLog/page?page=1&pageSize=10
Headers: Authorization: Bearer {token}

# 查询订单相关操作
GET /admin/operationLog/page?operationModule=订单
Headers: Authorization: Bearer {token}

# 查询某个用户的操作
GET /admin/operationLog/page?operatorId=1
Headers: Authorization: Bearer {token}

# 查询失败的操作
GET /admin/operationLog/page?status=0
Headers: Authorization: Bearer {token}

# 查询时间范围内的操作
GET /admin/operationLog/page?startTime=2024-12-01 00:00:00&endTime=2024-12-31 23:59:59
Headers: Authorization: Bearer {token}

# 超级管理员：查询特定家庭的操作日志
GET /admin/operationLog/page?familyId=1
Headers: Authorization: Bearer {super_admin_token}
```

## 最佳实践

### 1. 哪些操作需要记录日志？

**必须记录**：
- ✅ 关键业务操作（订单、支付、退款等）
- ✅ 管理员操作（增删改等）
- ✅ 用户登录/登出
- ✅ 权限变更
- ✅ 敏感数据访问
- ✅ 数据导入导出

**建议记录**：
- ✅ 状态变更
- ✅ 批量操作
- ✅ 审核操作

**不需要记录**：
- ❌ 普通查询（除非是敏感数据）
- ❌ 静态资源访问
- ❌ 健康检查接口

### 2. 参数记录建议

```java
// 推荐：记录参数，不记录结果（避免数据过大）
@OperationLog(
    operationType = OperationType.UPDATE,
    operationModule = "用户",
    operationDesc = "更新用户信息",
    recordParams = true,
    recordResult = false  // 结果通常不需要记录
)

// 特殊情况：需要追溯结果的操作
@OperationLog(
    operationType = OperationType.INSERT,
    operationModule = "订单",
    operationDesc = "提交订单",
    recordParams = true,
    recordResult = true  // 订单ID等关键信息
)
```

### 3. 敏感信息处理

操作日志会自动进行敏感信息脱敏：

```java
// 自动脱敏的信息
- 密码：password=****** 
- 手机号：138****5678
- Token：eyJh****
- 身份证号：123456********1234
- 邮箱：exa***@test.com
```

### 4. 日志清理建议

定期清理历史操作日志，避免数据库过大：

```sql
-- 清理90天前的日志
DELETE FROM operation_log WHERE create_time < DATE_SUB(NOW(), INTERVAL 90 DAY);

-- 或通过接口批量删除
DELETE /admin/operationLog/batch?endTime=2024-01-01 00:00:00
```

建议保留时长：
- 订单相关：180天
- 用户操作：90天
- 其他操作：30天

## 性能优化

### 1. 异步保存

操作日志使用异步线程保存，不会阻塞业务请求：

```java
// 在OperationLogAspect中
private void saveLogAsync(OperationLog operationLog) {
    new Thread(() -> {
        operationLogService.save(operationLog);
    }).start();
}
```

### 2. 数据限制

- 请求参数：最大2000字符，超出自动截断
- 响应结果：最大2000字符，超出自动截断
- 错误信息：最大500字符

### 3. 索引优化

已创建的索引：
- `idx_operator_id`：按操作人查询
- `idx_operation_type`：按操作类型查询
- `idx_operation_module`：按操作模块查询
- `idx_create_time`：按时间查询
- `idx_family_id`：按家庭ID查询

## 故障排查

### 1. 日志未记录

**检查点**：
1. 是否添加了 `@OperationLog` 注解？
2. 是否在 Controller 方法上？
3. 数据库表是否创建？
4. 检查应用日志是否有异常

### 2. 操作人信息为空

**原因**：请求未携带有效的Token或Token已过期

**解决**：确保请求头包含 `Authorization: Bearer {token}`

### 3. 日志数据过大

**优化**：
1. 设置 `recordParams = false` 不记录参数
2. 设置 `recordResult = false` 不记录结果
3. 定期清理历史日志
