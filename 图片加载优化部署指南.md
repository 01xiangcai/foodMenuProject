# 餐饮管理系统：图片加载优化部署指南

本文档记录了图片加载优化的核心改动以及线上生产环境的 [Nginx] 部署配置说明。

## 1. 核心改动说明

### 后端 (food_menu)
- **依赖引入**：添加 `net.coobird:thumbnailator:0.4.20`。
- **上传策略**：修改 `LocalStorageStrategy`，在保存文件时同时生成 400x400、80% 质量的 `_thumb` 后缀缩略图。
- **动态生成接口**：新增 `/common/dynamic-thumb` 接口（配合 Nginx 使用），支持对历史旧图片在请求时动态生成缩略图。
- **批量补齐工具**：新增 `/admin/migration/thumbnails/batch` 接口，支持扫描文件系统一键补齐所有历史图片的缩略图。

### 小程序端 (food_menu_uniapp)
- **工具函数**：重构 `getDishImage`，支持获取 `_thumb` 后缀。
- **列表页适配**：首页、分类菜单、收藏、订单列表等均已改为加载缩略图（`getDishImage(item, true)`）。
- **动效增强**：
  - 开启 `lazy-load`。
  - 增加全局 **骨架屏 (Skeleton Shimmer)** 动画。
  - 增加图片加载成功后的 **真·渐现过渡** 效果。

## 2. 线上 Nginx 配置 (核心部署)

为了实现老数据的平滑过渡，请在生产服务器的 Nginx 配置文件中参考以下 `try_files` 逻辑：

```nginx
# --- 餐饮系统图片存储配置 ---
# 假设图片物理存储路径为 E:/uploads/food-menu/
# 访问 URL 为 /uploads/...

location /uploads/ {
    # 1. 指向您的本地图片存放根目录
    alias E:/uploads/food-menu/; 
    
    # 2. 设置缓存头，提高性能
    expires 30d;
    
    # 3. 核心策略：
    #    - 先尝试访问 $uri (如果缩略图已生成，则直接秒开静态文件)
    #    - 如果文件不存在 (例如老图片还没有缩略图)，则转发给 @thumb_fallback 处理
    try_files $uri @thumb_fallback;
}

# 缩略图后备生成逻辑 (回源 Spring Boot)
location @thumb_fallback {
    # 转发给本地负载或指定的 Java 后端
    # Java 接口会读取原图 -> 生成缩略图落盘 -> 并返回图片流
    proxy_pass http://127.0.0.1:8080/common/dynamic-thumb?path=$uri;
    
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

## 3. 运维操作建议

1. **部署后端**：确保 `pom.xml` 和 `ThumbnailServiceImpl` 等代码已打包部署。
2. **部署前端**：重新编译小程序并发布。
3. **补齐缓存 (可选)**：如果您希望提高响应速度，可以登录后台（或通过 Postman）调用一次如下接口：
   - `POST /admin/migration/thumbnails/batch`
   - 此操作会遍历磁盘，为所有老图片补齐缩略图，后续 Nginx 访问将全部命中静态文件，不再走 Java 逻辑。
